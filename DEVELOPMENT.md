# Draw Tarot 開発方針

## 目的

問いをひとつ心に置いて、大アルカナを1枚引くための静的Webアプリ。
占い結果を断定するのではなく、ユーザが自分の問いを見る角度を変えるための道具として作る。

## 体験の優先順位

- スマホで気持ちよく使えることを最優先にする。
- 初期画面では、デッキ、問いの注意書き、操作ボタンだけに集中させる。
- 操作に直接必要ない説明は初期画面に置かない。
- ボタンは実機スマホの初期表示で確実に見えるようにする。
- カードの絵柄をよく見たいユーザが多い前提で、結果カードは大きく表示する。
- 結果の下に解説を置き、ユーザが自然にスクロールして読める流れにする。

## ドロー体験

- 注意書きはデッキ上に重ね、読みやすさのために薄い半透明背景と軽いぼかしを使う。
- 注意書きの例文は、3つ全部ではなく「問いの例」だと明確に分かる表現にする。
- シャッフル開始後、少し間を置いてから注意書きをゆっくり消す。
- カットは剣で切る演出ではなく、任意位置で上下を入れ替える操作として見せる。
- 引いたカードはデッキから下へ抜けるように動かす。
- 結果へのスクロールは、加速後にだんだん緩まって止まる感触にする。
- 結果へ到達したら、引いたカードはユーザに渡したものとして扱い、残りカードは揃えて一山に戻す。

## 表示と画像

- カード画像は切らず、全体が見えることを優先する。
- 画像差し替え時は、画像ファイル側で向きを正規化する。
- カード枠や表示比率は、実際の画像比率に合わせる。
- GitHub Pagesで動かす静的HTMLなので、不要なビルド工程や重い依存は増やさない。

## 解説テキスト

- 解説はカードの象徴、絵柄、キーワード、生命の樹上の位置づけとして書く。
- 「あなたはこうなる」のような占断文にはしない。
- 本文は後から差し替えやすい形を保つ。
- カード番号と名称も解説データ側で管理する。

## テストと運用

- 変更後は原則 `npm test` を実行する。
- Node v16由来のnpm警告が出ることがあるが、現状はテスト成功を優先してよい。
- コミット前に `git diff HEAD` を読み、差分からコミットメッセージを作る。
- ユーザの未コミット変更がある場合は、依頼対象だけをstageする。
- push前に `git status --short` で意図しない変更を確認する。
